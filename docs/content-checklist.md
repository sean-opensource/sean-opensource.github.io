# Chilepluto content checklist

Use this checklist before publishing or updating technical articles.

## Public-safe review

- [ ] Uses public-safe technical language.
- [ ] Avoids employer, client, internal system, and sensitive environment details.
- [ ] Avoids restricted or classification-specific wording.
- [ ] Removes secrets, tokens, tenant IDs, subscription IDs, hostnames, and internal URLs.
- [ ] Sanitises screenshots, logs, command output, and diagrams.

## Technical accuracy

- [ ] Commands were tested where practical.
- [ ] Version-sensitive guidance has been checked against current vendor documentation.
- [ ] Azure, GitHub Actions, Terraform, Python, and PowerShell examples state assumptions where needed.
- [ ] Limitations and trade-offs are included.
- [ ] Risky operations include rollback or safety notes.

## Structure

- [ ] Front matter includes title, date, categories, tags, and description.
- [ ] Article has a clear introduction and practical outcome.
- [ ] Headings are descriptive.
- [ ] Code blocks include language identifiers where useful.
- [ ] Long articles include checklists or summary sections.
- [ ] Related articles are linked at the end.

## SEO and discovery

- [ ] Description is specific and useful.
- [ ] Title matches the search intent.
- [ ] Internal links point to relevant hub and topic pages.
- [ ] External source links are current and authoritative.
- [ ] Images include descriptive filenames and useful alt text where supported.
- [ ] The article fits an existing topic cluster or introduces a clear new cluster.

## Maintenance

- [ ] Add a "Last reviewed" note for version-sensitive content.
- [ ] Add follow-up article ideas to the backlog.
- [ ] Check that scheduled publish date matches the content calendar.
- [ ] Confirm the article does not break the Monday/Friday publishing rhythm.
